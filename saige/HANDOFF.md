# Farm Advisory — Backend Architecture Handoff

> Single reference for Firestore (long-term persistence) and Redis (short-term memory / session).

**Key files:**

| Module | Purpose |
|--------|---------|
| `chat_history.py` | Firestore read/write helpers |
| `redis_client.py` | Redis connection manager with pooling |
| `message_buffer.py` | Last-N message buffer (Redis) |
| `graph.py` | LangGraph compilation + Redis checkpointing |
| `config.py` | All environment-driven configuration |
| `api.py` | FastAPI endpoints, request flow, rate limiter |

**Tests:**

| File | Covers |
|------|--------|

## 1. Identifiers

### `user_id`

| Property | Value |
|----------|-------|
| Type | `str` |
| Default | `"anonymous"` |
| Source | Sent by the frontend in every API request (`ChatRequest.user_id`). |
| Format | Free-form string. Currently always `"anonymous"`. When auth is added, replace with the authenticated user's ID (e.g. Firebase Auth UID). |

### `thread_id`

| Property | Value |
|----------|-------|
| Type | `str` |
| Source | Generated by the frontend on each new conversation. |
| Format | `thread_{unix_timestamp_ms}_{random_9_alphanum}` |
| Example | `thread_1708123456789_a3f9k2x8m` |
| Uniqueness | Guaranteed by timestamp + random suffix. |

## 2. Message Schema (shared by Firestore + Redis)

Canonical message shape used everywhere:

```json
{
  "role": "user | assistant | system | tool",
  "content": "message text",
  "ts": "2026-02-17T12:00:00.000000",
  "metadata": {
    "type": "quiz",
    "options": ["Option A", "Option B"],
    "advisory_type": "crops",
    "recommendations": ["Rec 1", "Rec 2"],
    "latency_ms": 1250
  }
}
```

- `role` — one of `user`, `assistant`, `system`, `tool`
- `content` — the message text
- `ts` — ISO-8601 timestamp (set on persist)
- `metadata` — optional map; allowed keys are controlled by `METADATA_ALLOWED_KEYS` in config

## 3. Firestore (Long-Term Persistence)

Implementation: `chat_history.py` · Config: `THREADS_COLLECTION` in `config.py`

### Schema

#### Collection: `threads/{thread_id}`

Thread metadata — one document per conversation.

| Field | Type | Description |
|-------|------|-------------|
| `thread_id` | `string` | Same as the document ID. |
| `user_id` | `string` | Owner of the thread. |
| `created_at` | `string` (ISO-8601) | When the first message was saved. |
| `updated_at` | `string` (ISO-8601) | Updated on every new message and on completion. |
| `status` | `string` | `"active"` or `"complete"`. |
| `advisory_type` | `string \| null` | `"weather"`, `"livestock"`, `"crops"`, `"mixed"`, or `null`. Set on completion. |
| `message_count` | `integer` | Total messages in the thread (incremented atomically). |
| `preview` | `string` | First user message, truncated to 100 chars. |
| `farm_context` | `map \| null` | Optional context set on completion (location, crops, farm_size, assessment_summary). |

**Example document** (`threads/thread_1708123456789_a3f9k2x8m`):

```json
{
  "thread_id": "thread_1708123456789_a3f9k2x8m",
  "user_id": "anonymous",
  "created_at": "2026-02-17T12:00:00.000000",
  "updated_at": "2026-02-17T12:05:30.000000",
  "status": "complete",
  "advisory_type": "crops",
  "message_count": 6,
  "preview": "my wheat has yellow leaves and the tips are drying",
  "farm_context": {
    "location": "Hayward, California",
    "crops": ["wheat"],
    "assessment_summary": "Wheat with yellowing leaves..."
  }
}
```

#### Subcollection: `threads/{thread_id}/messages/{message_id}`

Individual messages — one document per turn.

| Field | Type | Description |
|-------|------|-------------|
| `role` | `string` | `"user"`, `"assistant"`, `"system"`, or `"tool"`. |
| `content` | `string` | The message text. |
| `ts` | `string` (ISO-8601) | Timestamp when the message was persisted. |
| `metadata` | `map \| null` | Optional. May contain `latency_ms`, `advisory_type`, `recommendations`, `type` (`"quiz"`), `options`. |

**`message_id` format:** `{ISO-timestamp}_{random_8_hex}` — guarantees uniqueness and lexicographic ordering by time.

### Write Path

| Helper | Description |
|--------|-------------|
| `save_message(user_id, thread_id, role, content, metadata)` | Upserts thread doc + appends message to subcollection |
| `mark_complete(user_id, thread_id, advisory_type, farm_context)` | Sets `status="complete"` with optional metadata |

Error handling: failures return `False` and log the error — they never crash the request.

### Read APIs

#### `GET /threads?user_id=&limit=&cursor=`

List threads for a user, ordered by `updated_at` DESC.

```
threads
  .where("user_id", "==", <user_id>)
  .order_by("updated_at", DESCENDING)
  .start_after({"updated_at": <cursor>})
  .limit(<limit>)
```

Response: `{ "threads": [...], "next_cursor": "<updated_at>" | null }`

#### `GET /threads/{thread_id}/messages?user_id=&limit=&cursor=`

Load messages for a thread, ordered by `ts` ASC.

```
threads/<thread_id>/messages
  .order_by("ts")
  .start_after({"ts": <cursor>})
  .limit(<limit>)
```

Response: `{ "thread_id": "...", "messages": [...], "next_cursor": "<ts>" | null }`

#### `GET /analytics?user_id=`

Aggregate analytics across all threads (thread count, advisory type breakdown, latency stats).

### Required Firestore Indexes

| Collection | Fields | Order |
|------------|--------|-------|
| `threads` | `user_id` (ASC), `updated_at` (DESC) | Composite — required for thread listing |

Single-field indexes on `messages.ts` (ascending) are auto-created by Firestore.

### Health Check

**Endpoint:** `GET /health/firestore`

Performs a write → read → delete cycle on a `_health_check/test` document.

- `200 {"status": "healthy", "service": "firestore"}` — Firestore is accessible
- `503 {"status": "unhealthy", "service": "firestore"}` — Firestore is down or permissions are wrong

### Observability

In-process counters via `chat_history.get_metrics()`:

| Counter | Description |
|---------|-------------|
| `write_ok` | Successful message writes |
| `write_fail` | Failed message writes |
| `messages_logged` | Total messages persisted |

All Firestore operations emit structured log lines via `logging.getLogger("farm_advisory.chat_history")` with write latency in milliseconds.

## 4. Redis (Short-Term Memory + Session)

Implementation: `redis_client.py` + `message_buffer.py` · Config: `config.py`

### Provisioning

**Local dev** — `docker-compose.yml`:

```bash
docker compose up --build
```

Services: Backend (`http://localhost:8000`) + Redis (`localhost:6379`).
Compose sets `REDIS_ENABLED=true` and `REDIS_URL=redis://redis:6379/0`.
Redis data is persisted in volume `redis_data`.

**Production** — GCP Memorystore Redis:

1. Provision Redis in GCP Memorystore.
2. Create Serverless VPC connector for Cloud Run.
3. Set `REDIS_ENABLED=true` and `REDIS_URL=redis://10.0.0.5:6379/0`.
4. For TLS, use `rediss://` and set `REDIS_SSL_CERT_REQS` as needed.

### Message Buffer

Key format: `thread:{thread_id}:last_messages`

| Helper | Description |
|--------|-------------|
| `push_message(thread_id, message)` | LPUSH + LTRIM to last N |
| `get_last_n(thread_id, n)` | Returns ordered messages (oldest → newest) |
| `clear_thread(thread_id)` | Deletes the thread's buffer |

- Buffer size: `SHORT_TERM_N` (default 20)
- TTL: `SHORT_TERM_TTL_SECONDS` (default 86400 = 24h)
- Content truncated to `MAX_STORED_CONTENT_CHARS` (2000) before storage
- Metadata filtered to `METADATA_ALLOWED_KEYS` whitelist, capped at `MAX_METADATA_BYTES` (2048)

### LangGraph Checkpointing

`graph.py` compiles the graph with `RedisSaver` when Redis is available:

```python
from langgraph.checkpoint.redis import RedisSaver
checkpointer = RedisSaver(redis_url)
checkpointer.setup()  # ensure RediSearch indexes exist
graph = builder.compile(checkpointer=checkpointer)
```

Falls back to `MemorySaver` if:
- `langgraph-checkpoint-redis` is not installed
- Redis connection fails

`api.py` has `safe_graph_stream()` which catches missing checkpoint index errors mid-stream and transparently falls back to `MemorySaver` for that request.

### Health Check

**Endpoint:** `GET /health/redis`

- `200 {"status": "disabled"}` — Redis intentionally turned off
- `200 {"status": "healthy"}` — Redis is reachable (includes latency)
- `503 {"status": "unhealthy"}` — Redis enabled but unreachable

**Quick verification:**

```bash
python test_redis.py
```

### Troubleshooting

1. **Redis unavailable** — check `REDIS_URL` or fallback host/port values; verify Redis is reachable from runtime network.
2. **Health endpoint returns 503** — check backend logs for Redis connection errors; validate VPC connector route in Cloud Run (production).
3. **TLS issues** — verify `rediss://` scheme and `REDIS_SSL_CERT_REQS` setting.

## 5. Safety Controls

| Control | Config Var | Default | Description |
|---------|-----------|---------|-------------|
| Input size cap | `MAX_MESSAGE_CHARS` | 4000 | Server-side hard cap on user input length |
| Storage truncation | `MAX_STORED_CONTENT_CHARS` | 2000 | Messages truncated before Redis storage |
| Metadata whitelist | `METADATA_ALLOWED_KEYS` | `{type, options, advisory_type, recommendations}` | Only these keys are stored |
| Metadata size cap | `MAX_METADATA_BYTES` | 2048 | Max serialized size for metadata dict |
| Rate limit toggle | `RATE_LIMIT_ENABLED` | `true` | Enable/disable per-thread rate limiting |
| Rate limit requests | `RATE_LIMIT_MAX_REQUESTS` | 20 | Max requests per window |
| Rate limit window | `RATE_LIMIT_WINDOW_SECONDS` | 60 | Window duration in seconds |

Rate limiting uses Redis `INCR` + `EXPIRE` per thread. Fails open (allows request) if Redis is unavailable.

## 6. Request Flow

```
User input
  │
  ▼
Rate limit check (Redis INCR + EXPIRE)
  │ 429 if exceeded
  ▼
Read last-N from Redis buffer → build short_term_history
  │
  ▼
Save user message to Firestore (long-term) + Redis buffer (short-term)
  │
  ▼
graph.stream() with Redis checkpointing
  │ (falls back to MemorySaver if Redis indexes missing)
  ▼
┌─────────────────────────────────┐
│  Assessment → Routing → Advisory │
│  (LangGraph nodes)               │
└─────────────────────────────────┘
  │
  ├─ requires_input → Save quiz to Firestore + Redis → return UI schema
  │
  └─ complete → Save diagnosis to Firestore + Redis
                 → mark_complete with farm_context
                 → return response
```
